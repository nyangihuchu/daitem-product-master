## 1. 프로젝트 개요

### 1.1 서비스 성격

공구 유통 사업의 **내부 운영 전용 백오피스(Back-office) 웹서비스**.  
공급처 수급 상품 12만 개의 관리를 시작으로, 주문·발주·판매·수익 분석까지 단일 플랫폼에서 처리한다.

| 항목 | 내용 |
|------|------|
| 서비스 성격 | 공구 유통 사업 전용 내부 운영 백오피스 (내부 사용자 전용) |
| 상품 규모 | 공급처 수급 12만 개 / 카페24 등록 8만 개 |
| 판매 채널 | 자사몰(카페24) + 네이버스토어, 쿠팡, 지마켓, 옥션, 롯데온, 11번가 (총 7개 채널) |
| 개발 방식 | Claude Code 활용 AI 보조 개발 |
| 확장 목표 | 카페24 API 연동 → 멀티마켓 자동화 → 발주 자동화 |

### 1.2 개발 배경 및 필요성

| 현황 (문제) | 필요성 |
|-------------|--------|
| 공급처 상품 12만 개 중 카페24 미등록 상품이 4만 개에 달함 | 전체 상품의 통합 현황 파악 및 등록 우선순위 관리 체계 |
| 7개 판매 채널의 상품 등록 현황을 한 곳에서 파악 불가 | 멀티채널 등록 현황 단일 뷰 |
| 채널별 수수료를 반영한 실질 마진율 계산 체계 부재 | 채널별 수수료 반영 수익 분석 기능 |
| 주문 발생 시 공급처 발주까지의 프로세스가 수작업으로만 운영 | 주문-발주 연계 처리 및 자동화 기반 마련 |
| 팀 내 일정 공유 및 알림 체계 미비 | 팀 협업 일정 관리 및 자동 알림 플랫폼 |

## 2. 기술 스택

| 영역 | 기술 | 선정 이유 |
|------|------|-----------|
| **풀스택 프레임워크** | Next.js (App Router) | 프론트 + API Routes 통합, Supabase 궁합 최적, Vercel 배포 용이 |
| **데이터베이스** | Supabase (PostgreSQL) | 실시간 DB·Auth·Storage 내장, 무료 티어로 시작 가능 |
| **UI 프레임워크** | React + Tailwind CSS + shadcn/ui | 컴포넌트 재사용 용이, 대용량 테이블 처리 최적 |
| **검색** | Supabase Full-text Search | 별도 엔진 불필요, 한국어 전문 검색 내장 |
| **엑셀 처리** | Next.js API Routes + xlsx 라이브러리 | 대용량 CSV/XLSX 스트리밍 업로드, Supabase bulk insert 연계 |
| **호스팅** | Vercel (프론트) + Supabase (백엔드·DB) | Next.js-Vercel 최적 궁합, 초기 비용 최소화 |

## 3. 데이터 모델

초기 설계에 반드시 포함되어야 할 핵심 테이블 목록.  
**카페24 연동(Phase 4) 대비 매핑 컬럼을 Phase 1부터 포함할 것.**

```
suppliers         — 공급처 마스터 (코드, 담당자, 계약조건, 리드타임)
products          — 상품 마스터 (SKU, 내부코드, 분류, 가격이력, 상태, cafe24_product_id)
market_listings   — 마켓별 상품 등록 현황 (마켓명, 상품코드, URL, 등록일)
price_history     — 구입가 변경 이력 (상품ID, 변경일, 이전가, 변경가)
orders            — 주문 마스터 (채널, 주문번호, 상태, 고객정보)
order_items       — 주문 상품 상세 (주문ID, 상품ID, 수량, 판매가)
purchases         — 발주 마스터 (공급처, 발주일, 상태, 총액)
purchase_items    — 발주 상품 상세 (발주ID, 상품ID, 수량, 매입가)
market_fees       — 마켓별 수수료율 설정 (마켓명, 수수료율, 적용일) ← 변동 대응
schedules         — 일정 마스터 (유형, 담당자, 날짜, 알림 설정)
users             — 사용자 (역할, 권한, 알림 수신 설정)
```

> **설계 원칙**
> - `products.cafe24_product_id` 컬럼을 Phase 1부터 포함 (Phase 4 연동 대비)
> - 매입가·정산 데이터는 소프트 딜리트(soft delete)만 허용, 물리 삭제 금지 (회계 기록)
> - 마켓 수수료율은 `market_fees` 테이블로 분리해 변동 시 이력 관리

## 4. 기능 명세

### 4.1 상품 관리 (Product Management)

공급처로부터 수급한 12만 개 상품의 등록·관리·검색 및 멀티채널 등록 현황을 통합 관리한다.

#### 상품 데이터 구조
- 공급처 상품코드(SKU) + 내부 관리코드 **이원화** 관리
- 공급처 마스터 테이블 별도 구성 (다수 공급처 대응)
- 상품 단위(개/세트/박스), 최소발주수량(MOQ), 브랜드 필드 포함
- 상품 분류 체계: **대분류 / 중분류 / 소분류** (전동공구·수공구·측정기기 등)

#### 상품 상태
| 상태값 | 설명 |
|--------|------|
| `판매중` | 정상 판매 |
| `품절` | 재고 없음 |
| `단종` | 공급처 단종 |
| `등록대기` | 마켓 등록 전 |
| `검토중` | 판매 여부 검토 |

- 마켓별 등록 여부: O/X가 아닌 **마켓별 상품코드 + 등록 URL** 함께 저장
- 공급처 단종 상품 자동 상태 변경 처리

#### 가격 관리
- 구입가(매입가) **변경 이력** 관리 — 공급처 가격 변동 추적
- **채널별 판매가 독립 관리** (자사몰 / 쿠팡 / 네이버 등 채널마다 다른 가격 설정)
- 권장소비자가(MSRP) 필드 — 할인율 표시 마케팅 활용

#### 재고 관리
- 재고 하한선 알림 — 설정 수량 이하 시 담당자 알림 발송
- 공급처 보유 재고 vs 내부 보유 재고 구분
- 재고 변동 이력 추적

#### 검색 및 일괄 처리
- 다중 조건 필터: 카테고리, 브랜드, 공급처, 가격대, 마진율 범위, 마켓 등록 여부
- **엑셀(CSV/XLSX) 일괄 업로드 / 다운로드** — 12만 개 대량 처리 필수
- 키워드 전문 검색 (Supabase Full-text Search)

---

### 4.2 주문·발주 관리 (Order & Purchase Management)

자사몰 및 각 마켓의 주문을 접수하고, 공급처에 발주 처리하는 전체 흐름을 관리한다.  
**초기: 수동 입력 / Phase 4: API 자동 수집으로 확장**

#### 주문 처리 흐름

```
주문 접수 → 시스템 입력 → 발주서 생성 → 공급처 전달 → 배송 정보 입력 → 완료 처리
 (STEP 1)     (STEP 2)      (STEP 3)       (STEP 4)        (STEP 5)         (STEP 6)
```

#### 주문 상태값

```
주문접수 → 발주완료 → 배송중 → 배송완료 → 정산완료
                   ↘ 취소 / 반품 / 교환 (별도 상태)
```

- 마켓별 주문번호 + 내부 주문번호 **이중 관리**

#### 발주서 기능
- 발주서 자동 생성 및 **PDF 출력 / 이메일 발송**
- 공급처별 발주 양식 **템플릿** 관리 (공급처마다 양식 상이)
- 동일 공급처 복수 상품 **일괄 발주** 묶음 처리

#### 멀티채널 주문 통합
- 자사몰 외 7개 마켓 주문 통합 관리
- 배송 추적: 운송장 번호 → 배송 상태 조회 (스마트택배 API 연동)

---

### 4.3 판매·수익 관리 (Sales & Margin Management)

매입가, 채널별 수수료, 배송비 등을 반영한 **실질 마진율**을 계산하고 판매 현황을 분석한다.  
**초기: 수동 입력 / Phase 3: 자동 집계로 확장**

#### 실질 마진 계산 구조

```
실질 마진 = 판매가 - 매입가 - 마켓수수료 - 배송비 - (반품비용) - (광고비)
```

| 비용 항목 | 내용 | 비고 |
|-----------|------|------|
| 매입가 | 공급처 구입 원가 | 필수 |
| 마켓 수수료 | 쿠팡 ~10%, 네이버 2~5% 등 | 채널별 상이 |
| 배송비 | 무료배송 정책 시 원가 산입 | 정책에 따라 |
| 반품·교환 비용 | 비율 추정 적용 | 선택 |
| 광고비 | 마켓 내 광고 집행 시 | 선택 |

#### 판매 분석 기능
- 기간별 매출·매입·이익 집계: **일간 / 주간 / 월간 / 분기 / 연간**
- 채널별 매출 비중 분석 차트
- 상품별·카테고리별·브랜드별 판매 순위
- 잘 팔리는 상품 / 재고 정리 필요 상품 자동 분류

#### 정산 관리
- 마켓별 정산 주기 관리 (쿠팡 주 1회, 네이버 월 2회 등)
- 정산 예정일 + 정산 예상 금액 추적
- 미정산 잔액 현황 대시보드

### 4.4 일정·협업 관리 (Schedule & Collaboration)

팀 내 일정을 공유하고 **카카오 알림톡·이메일·SMS**로 자동 알림을 발송한다.

#### 일정 유형
| 유형 | 예시 |
|------|------|
| 공급처 관련 | 가격 변경 예정일, 신상품 입고일, 담당자 미팅 |
| 마켓 관련 | 행사 참여 일정, 판촉 기간, 광고 캠페인 |
| 내부 업무 | 재고 실사, 상품 등록 일정, 정산 확인일 |
| 고객 관련 | 클레임 처리 기한, 반품 처리 기한 |

#### 알림 채널 (우선순위 순)
1. **카카오 알림톡** — 한국 비즈니스 환경 최우선
2. **이메일 알림**
3. **브라우저 푸시 알림**
4. **SMS** — 긴급 공지

#### 사용자 권한
| 역할 | 권한 |
|------|------|
| 관리자 | 전체 기능 + 사용자 관리 |
| 운영자 | 상품·주문·발주·판매 관리 |
| 조회 전용 | 읽기만 가능 |

### 4.5 공급처 관리 (Supplier Management)

상품 관리와 연계되는 공급처 정보를 별도 마스터로 관리한다.

- 공급처별 담당자 정보, 연락처, 계약 조건 관리
- 리드타임(발주 후 배송까지 소요 시간) 기록
- 공급처 평가 이력: 배송 지연율, 불량률, 클레임 발생 횟수
- 공급처로부터 수신한 상품 목록 업데이트 이력 관리

### 4.6 메인 대시보드 (Dashboard)

로그인 진입 시 핵심 운영 지표를 한눈에 파악할 수 있는 통합 현황판.

**좌측 패널**
- 오늘 주문 건수 / 미처리 발주 건수
- 이번 달 매출 / 이익 요약
- 채널별 매출 비중 차트

**우측 패널**
- 재고 부족 상품 알림
- 처리 필요 반품·교환 건수
- D-day 임박 일정 목록

## 5. 개발 로드맵

| 단계 | 기간 | 주요 기능 | 상태 |
|------|------|-----------|------|
| **Phase 1** | 1~2주 | 상품 DB 구축, 검색·필터, 대시보드 | 즉시 착수 |
| **Phase 2** | 2~3주 | 주문·발주 관리, 마진 계산, 일정 관리 | 핵심 운영 |
| **Phase 3** | 3~4주 | 채널별 수익 분석, 정산 관리, 알림 자동화 | 분석 고도화 |
| **Phase 4** | 2~4주 | 카페24 API 연동, 마켓 주문 자동 수집, 발주 자동화 | 연동 자동화 |

### Phase 4 카페24 연동 사전 설계 (초기부터 반영)

- `products.cafe24_product_id` 컬럼 Phase 1 테이블에 포함
- 카페24 REST API 상품 정보 양방향 동기화 구조
- 주문 자동 수집 → 발주 자동화 흐름
- 마켓 7개 채널 주문 자동 수집 (네이버, 쿠팡, 지마켓, 옥션, 롯데온, 11번가)

## 6. 개발 주의사항

### 필수 사항

- **12만 개 상품 초기 적재**: 엑셀 일괄 업로드 → Next.js API Route 스트리밍 + Supabase bulk insert 처리 필수
- **마켓 수수료율**: 변동 가능하므로 `market_fees` 테이블로 분리, 이력 관리
- **카페24 연동 대비**: `products.cafe24_product_id` 컬럼을 Phase 1부터 포함
- **데이터 삭제 정책**: 매입가·정산 데이터는 회계 기록이므로 물리 삭제 금지 → 소프트 딜리트만 허용
- **모바일 반응형 UI**: 외부 이동 중 주문 현황 확인 가능하도록 반응형 필수

### 성능 고려사항

- 상품 목록 페이지: 12만 개 대응 → **페이지네이션 + 인덱스 최적화** 필수
- 검색: Supabase Full-text Search 한국어 형태소 분석기 설정
- 대용량 엑셀 업로드: 청크(chunk) 단위 분할 처리

### 확장성 고려사항

- API Routes는 카페24 연동 확장을 고려해 RESTful 구조로 설계
- 알림 발송(카카오/SMS)은 서비스 레이어로 분리해 채널 교체 용이하게 구성
- 마켓 채널 추가 시 `market_listings` 테이블만 확장하면 되도록 설계
